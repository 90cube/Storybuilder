# 스토리빌더 (Storybuilder)

> 스키마 주도(schema-driven) 스토리 데이터 편집기

캐릭터·장소·아이템·사건 같은 **스토리 자산**을 정해진 규칙(스키마)에 따라 입력·검증·저장하는 로컬 편집 도구입니다. 입력칸을 채우면 즉시 DB에 기록되고, 타입마다 필수 항목이 채워지지 않으면 저장이 막힙니다. 관계를 입력하면 상대 엔티티에도 자동으로 반영되어 데이터의 일관성이 구조적으로 유지됩니다.

도구(편집기)와 데이터(DB 파일)는 분리됩니다 — 스프레드시트 프로그램과 문서 파일의 관계와 같습니다.

---

## 핵심 특징

- **스키마 주도** — 타입·필드·필수 규칙을 *데이터 파일*로 정의합니다. 새 타입을 추가할 때 코드를 고치지 않고 정의 파일(JSON)만 늘리면 폼·검증·DB 구조가 자동으로 따라옵니다.
- **필수 입력 검증** — 타입별 필수 항목이 비어 있으면 저장이 잠깁니다(클라이언트·서버 양쪽 검증).
- **관계 양방향 주입** — `A —제자→ B`를 넣으면 `B —스승→ A`가 자동 생성됩니다(같은 쌍으로 묶임). 상대를 직접 건드리지 않아도 양쪽에 반영됩니다.
- **시계열 누적(타임라인)** — 엔티티별 상태 변화를 시간 순서로 쌓고 정렬해 조회합니다.
- **비밀/인지상태** — 어떤 사실을 누가·언제부터 아는지, 독자에게는 언제 공개되는지를 기록합니다. 새 인물을 끼워 넣을 때 앞뒤가 어긋나는 누설을 방지합니다.
- **동시편집 안전** — 버전 기반 낙관적 잠금으로 다른 사용자가 먼저 수정한 경우 덮어쓰기를 차단합니다.
- **편집 로그** — 모든 생성·수정·삭제(누가·언제·무엇을·전→후)를 자동 기록합니다.
- **PIN 인증** — 사번 기반 ID + PIN 로그인. 최초 로그인 시 PIN 강제 변경(이중 입력 일치). PIN은 해시로만 저장합니다.
- **엔진 독립** — 표준 SQL만 사용해 SQLite(휴대)·MySQL(팀 동시편집) 같은 코드로 동작합니다.
- **내보내기** — 내용 테이블을 JSON·CSV 텍스트로 내보내 버전관리·공유하고, 데이터 웨어하우스 적재 스크립트도 생성합니다.

## 데이터 모델

| 구분 | 테이블 | 역할 |
|------|--------|------|
| 내용 | `entities` | 엔티티(행 1개 = 캐릭터/장소/아이템…). 타입별 필드는 `data`(JSON 컬럼)에 저장 |
| 내용 | `relations` | 관계(정/역 양방향, 같은 `pair_id`로 묶임) |
| 내용 | `timeline` | 엔티티별 상태 스냅샷(시계열 누적) |
| 내용 | `secrets` | 비밀과 인지상태(누가 아는가/언제 공개되는가) |
| 시스템 | `sys_edit_log` | 편집 로그(자동 기록) |
| 시스템 | `sys_users` | 로그인 계정(PIN 해시) |
| 시스템 | `sys_schema_meta` | 스키마 정의 캐시(타입·필드·필수) |

**조립식 설계** — 엔티티 하나는 *공통 필드 + 타입 블록 + 선택 부품(관계·타임라인·비밀)* 을 합쳐 구성됩니다. 화면에서는 한 장(카드)으로 보이고, 저장은 효율을 위해 여러 행으로 나뉩니다.

## 프로젝트 구조

```
editor/
├─ schema/            스키마 정의 (메타데이터)
│  ├─ _base.json         공통 필드
│  ├─ _relations.json    관계 역방향 레지스트리
│  └─ <type>.json        타입별 정의 (예: character.json)
├─ backend/
│  ├─ schemadef/      정의 로드 (타입·필드·관계)
│  ├─ ddl/            정의 → 방언별 테이블 DDL 생성
│  ├─ store/          DB 연결 + 스키마 초기화
│  ├─ entity/         엔티티 CRUD·검증·버전잠금·관계·타임라인·비밀·편집로그
│  ├─ auth/           PIN 인증
│  ├─ migrate/        기존 데이터 이관(ETL)
│  ├─ export/         JSON/CSV 내보내기 + 적재 스크립트
│  └─ server/         웹 GUI (HTTP/JSON + 정적 페이지)
└─ cmd/
   ├─ schema-init/    빈 DB에 스키마 생성
   ├─ migrate/        기존 데이터 → DB 이관
   ├─ serve/          편집기 웹 GUI 기동
   └─ export/         DB → 텍스트 내보내기
```

## 빠른 시작

요구 사항: **Go 1.22+** (테스트·로컬 DB는 순수 Go SQLite 드라이버 사용 — C 컴파일러 불필요)

```bash
cd editor

# 1) 테스트
go test ./...

# 2) 빈 DB에 스키마 생성
go run ./cmd/schema-init -dialect sqlite -dsn ./data.db -schema ./schema

# 3) 편집기 GUI 실행 → 브라우저로 http://127.0.0.1:8765
go run ./cmd/serve -dsn ./data.db -schema ./schema

# 4) 데이터 내보내기 (JSON/CSV + 적재 스크립트)
go run ./cmd/export -dsn ./data.db -out ./export
```

빌드해서 단일 실행 파일로 배포할 수도 있습니다:

```bash
go build -o storybuilder-serve.exe ./cmd/serve
```

### 새 타입 추가

`editor/schema/`에 정의 파일 하나만 추가하면 됩니다(코드 수정 없음):

```json
{
  "type": "location",
  "label": "장소",
  "fields": [
    { "key": "summary",       "label": "한줄요약", "datatype": "text",   "required": true },
    { "key": "parent_region", "label": "상위지역", "datatype": "string", "required": false }
  ],
  "mixins": ["relations", "timeline"]
}
```

`schema-init`을 다시 실행하면 폼·검증·메타데이터에 자동 반영됩니다. 타입별 필드는 JSON 컬럼에 저장되므로 테이블 구조(DDL)는 바뀌지 않습니다.

## 기술 스택

- **언어:** Go (표준 라이브러리 중심 — `database/sql`, `net/http`, `embed`, `encoding/json`)
- **DB 드라이버:** `modernc.org/sqlite`(순수 Go, 로컬·테스트), `go-sql-driver/mysql`(팀)
- **인증:** `golang.org/x/crypto/bcrypt`
- **프론트:** 단일 HTML 페이지(바닐라 JS) — 빌드 도구 불필요, 실행 파일에 내장

---

## 설계 원칙

1. **스키마 주도** — 타입·필드·필수 규칙은 데이터로 정의하고, 폼·검증·DDL·이관이 모두 그 한 곳을 참조한다.
2. **엔진 독립** — 특정 DB 전용 기능을 피하고 표준 SQL만 사용한다.
3. **무결성은 구조로** — 필수 입력·관계 양방향·버전 충돌·편집 로그를 앱의 신뢰가 아니라 데이터 구조로 보장한다.
4. **도구와 데이터 분리** — 편집기는 가볍게 들고 다니고, 데이터는 별도 파일/서버로 관리한다.
